# SMTP の暗号化方式

## 平文 SMTP (ポート25)

元々の SMTP は暗号化なし。メール内容もパスワードもそのまま流れる。今でもサーバ間の中継に使われるが、クライアントからの送信には使うべきではない。

## Implicit TLS (ポート465)

```
クライアント ──── TLSハンドシェイク ────→ サーバ (465)
              （ここで暗号化が確立）
クライアント ←── TLS確立 ──────────────→ サーバ
クライアント ──── EHLO ────────────────→ サーバ
クライアント ──── AUTH ────────────────→ サーバ
              ... 以降すべて暗号化済み ...
```

**接続した瞬間から TLS**。HTTPS と同じ考え方。接続 = 即暗号化なので、平文が一切流れない。

コード上は `tls.Dial()` で接続して、その上で SMTP を喋る。

## STARTTLS (ポート587)

```
クライアント ──── 平文で TCP 接続 ─────→ サーバ (587)
クライアント ──── EHLO ────────────────→ サーバ
クライアント ←── 250-STARTTLS ────────── サーバ（「TLS対応してるよ」）
クライアント ──── STARTTLS ────────────→ サーバ（「TLS始めたい」）
クライアント ←── 220 Go ahead ────────── サーバ（「どうぞ」）
              （ここで TLS ハンドシェイク）
クライアント ──── AUTH ────────────────→ サーバ
              ... 以降すべて暗号化済み ...
```

**最初は平文で接続し、途中から TLS にアップグレード**する。`STARTTLS` というSMTPコマンドを送ることで「ここから暗号化に切り替えよう」と合意する。

コード上は `smtp.Dial()`（平文接続）→ `client.StartTLS(tlsConfig)` でアップグレード。

## 比較

| | Implicit TLS (465) | STARTTLS (587) |
|---|---|---|
| 接続開始 | 最初から暗号化 | 平文で始まる |
| 暗号化タイミング | TCP接続時 | SMTPコマンドで切り替え |
| 平文が流れる区間 | なし | EHLO〜STARTTLS の間 |
| 主な用途 | メール送信 (submission) | メール送信 (submission) |

どちらも最終的に暗号化された上で認証・送信するので、セキュリティ上の大きな差はない。どちらを使うかは接続先のメールサーバが何をサポートしているか次第。Gmail や多くのプロバイダは両方対応している。

## このプロジェクトでの対応

```go
// Config で指定
UseTLS: true        // → Implicit TLS (465) で接続
UseSTARTTLS: true   // → STARTTLS (587) で接続
// どちらも指定なし → 平文 (非推奨、今のコードでは STARTTLS のパスに入る)
```

`UseTLS` も `UseSTARTTLS` も `false` の場合、`sendWithSTARTTLS` に入るが、サーバが STARTTLS を提供しなければ平文のまま認証が走る可能性がある。
